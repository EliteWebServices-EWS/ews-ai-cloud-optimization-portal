# Frontend AWS Hosting Architecture

**Project:** EWS AI Cloud Optimization Platform (SISU'M)
**Organization:** Elite Web Services
**Status:** Planned
**Target AWS Region:** `us-east-1`

---

# 1. Purpose

Sprint 10.5.7 moves the EWS frontend from temporary GitHub Pages hosting to a secure and scalable AWS hosting architecture.

The frontend will be built as static files and hosted in a private Amazon S3 bucket.

Amazon CloudFront will provide:

* HTTPS
* Global content delivery
* Caching
* Origin protection
* Single-page application routing
* A future connection to the custom EWS domain

The backend remains independently deployed through:

```text
API Gateway
    ↓
AWS Lambda
    ↓
Express
```

The frontend and backend will not share the same deployment workflow.

---

# 2. Sprint Objective

Deploy the production frontend to AWS using:

```text
Amazon S3
    +
Amazon CloudFront
    +
CloudFront Origin Access Control
```

The frontend must:

* Build successfully using the existing frontend build system.
* Be stored in a private S3 bucket.
* Be accessible only through CloudFront.
* Load successfully through the CloudFront HTTPS domain.
* Support direct navigation to frontend application routes.
* Be prepared to call the live SISU'M backend.
* Remain separate from the backend SAM stack and backend workflow.
* Be ready for frontend CI/CD automation in the next sprint.

# 5. Architecture Principles

## 5.1 Private S3 Bucket

The frontend S3 bucket must not be publicly accessible.

The following S3 Block Public Access settings must remain enabled:

```text
BlockPublicAcls
IgnorePublicAcls
BlockPublicPolicy
RestrictPublicBuckets
```

The bucket must not use a public website bucket policy.

The frontend must not be accessed through an S3 website endpoint.

---

## 5.2 CloudFront as the Public Entry Point

CloudFront will be the only public frontend entry point.

Users will access a URL similar to:

```text
https://dxxxxxxxxxxxxx.cloudfront.net
```

CloudFront will retrieve frontend objects from the private S3 bucket.

---

## 5.3 Origin Access Control

CloudFront Origin Access Control will authorize CloudFront to read objects from the private S3 bucket.

The architecture must use:

```text
Origin Access Control
```

and not the older:

```text
Origin Access Identity
```

The CloudFront origin must be the normal S3 bucket origin, not the S3 static website endpoint.

---

## 5.4 Independent Deployments

Frontend and backend deployments must remain independent.

Backend changes must not redeploy the frontend.

Frontend changes must not redeploy the backend.

Target workflow structure:

```text
.github/workflows/
├── deploy-backend.yml
└── deploy-frontend.yml
```

`deploy-frontend.yml` will be implemented in Sprint 10.5.8.

---

## 5.5 Infrastructure Separation

The backend infrastructure is managed through:

```text
backend/template.yaml
AWS SAM
CloudFormation stack: sisum-backend
```

Frontend infrastructure must use its own stack or infrastructure definition.

Recommended frontend stack name:

```text
sisum-frontend-production
```

The frontend must not be added to the existing backend SAM stack.

# 7. S3 Bucket Policy Design

The S3 bucket policy must allow only the CloudFront service to read frontend files.

The policy must use the CloudFront distribution ARN as a condition.

Target structure:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontReadOnly",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::FRONTEND_BUCKET_NAME/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::739275446782:distribution/DISTRIBUTION_ID"
        }
      }
    }
  ]
}
```

The placeholders must be replaced with actual deployed values.

The policy must not contain:

```json
"Principal": "*"
```

# 9. Frontend Environment Configuration

The frontend must not hardcode localhost URLs in production.

The production API base URL should be supplied through a Vite environment variable.

Recommended variable:

```text
VITE_API_BASE_URL
```

Production value:

```text
https://zqe6cl0m15.execute-api.us-east-1.amazonaws.com
```

Example production environment file:

```text
frontend/.env.production
```

Example content:

```env
VITE_API_BASE_URL=https://zqe6cl0m15.execute-api.us-east-1.amazonaws.com
```

The frontend application should read it using:

```ts
const apiBaseUrl = import.meta.env.VITE_API_BASE_URL;
```

No AWS credentials, secrets, tokens or passwords may be stored in a Vite environment variable.

All variables prefixed with `VITE_` are treated as frontend build-time values and can be visible to browser users.

# 10. Backend CORS Dependency

The backend currently allows this frontend origin:

```text
https://elitewebservices-ews.github.io
```

The first CloudFront deployment will create a domain similar to:

```text
https://dxxxxxxxxxxxxx.cloudfront.net
```

The backend CORS configuration must be updated before browser integration testing.

The backend may temporarily support both origins during migration:

```text
https://elitewebservices-ews.github.io
https://dxxxxxxxxxxxxx.cloudfront.net
```

The exact implementation depends on whether the backend currently supports one CORS origin or an array of origins.

The backend must not be changed blindly.

The existing Express and SAM CORS implementation must be inspected first.

After DNS migration, the final production origin should become:

```text
https://elitewebservices.org
```

and possibly:

```text
https://www.elitewebservices.org
```

Temporary GitHub Pages and CloudFront origins should be removed after migration is complete.

# 11. Single-Page Application Routing

The frontend uses client-side routing or may use it in future.

A user may directly request a route such as:

```text
/dashboard
/reports
/governance
/financial
```

Those paths may not exist as physical files in S3.

CloudFront must return:

```text
/index.html
```

for relevant origin errors.

Recommended custom error responses:

## Error Response 1

| Setting                   | Value         |
| ------------------------- | ------------- |
| Origin status             | 403           |
| Response page path        | `/index.html` |
| HTTP response code        | 200           |
| Error caching minimum TTL | 0             |

## Error Response 2

| Setting                   | Value         |
| ------------------------- | ------------- |
| Origin status             | 404           |
| Response page path        | `/index.html` |
| HTTP response code        | 200           |
| Error caching minimum TTL | 0             |

This allows the browser application router to handle frontend routes.

API errors must continue to come directly from API Gateway and must not be rewritten by the frontend CloudFront distribution.

# 12. Cache Strategy

Frontend files need different caching behavior.

## HTML

`index.html` should use a short cache duration or no-cache behavior so new deployments become visible quickly.

Recommended metadata:

```text
Cache-Control: no-cache, no-store, must-revalidate
```

## Hashed JavaScript and CSS Assets

Vite normally generates hashed filenames such as:

```text
assets/index-a1b2c3.js
assets/index-d4e5f6.css
```

These files may use long-lived caching:

```text
Cache-Control: public, max-age=31536000, immutable
```

## CloudFront Invalidation

For the first manual deployment, invalidate:

```text
/*
```

Later automated deployments may invalidate:

```text
/index.html
```

while allowing immutable hashed assets to remain cached.

# 13. HTTPS and Certificate Strategy

## Phase 1 — CloudFront Domain

The first deployment may use the default CloudFront certificate and domain:

```text
https://dxxxxxxxxxxxxx.cloudfront.net
```

This allows the AWS frontend to be tested before changing public DNS.

## Phase 2 — Custom Domain

For:

```text
elitewebservices.org
www.elitewebservices.org
```

an AWS Certificate Manager public certificate will be required.

The certificate used by CloudFront must be requested in:

```text
us-east-1
```

Recommended certificate names:

```text
elitewebservices.org
www.elitewebservices.org
```

DNS validation records will be created at the current DNS provider.

Custom-domain DNS migration should occur only after the CloudFront URL has passed all tests.

# 19. Frontend IAM Strategy

A dedicated frontend deployment role should be used.

Recommended role name:

```text
SisumFrontendDeployRole
```

It should not reuse:

```text
SisumBackendDeployRole
```

The frontend role will eventually require narrowly scoped permissions for:

* Uploading and deleting objects in the frontend S3 bucket
* Reading bucket metadata
* Creating CloudFront invalidations
* Reading the target CloudFront distribution
* Deploying the frontend infrastructure stack if CI/CD owns infrastructure

The role should not receive:

```text
AdministratorAccess
Lambda permissions
API Gateway write permissions
IAM role-management permissions
Access to unrelated S3 buckets
```

# 20. Security Controls

Sprint 10.5.7 must implement the following controls:

* S3 Block Public Access remains enabled.
* S3 ACLs remain disabled.
* S3 Object Ownership is bucket-owner enforced.
* CloudFront uses OAC.
* Bucket policy restricts access to one CloudFront distribution.
* Viewer requests redirect from HTTP to HTTPS.
* No AWS credentials are placed in frontend files.
* No backend secrets are exposed to the frontend.
* Frontend and backend IAM roles remain separate.
* Frontend and backend CloudFormation stacks remain separate.
* Production API stays in Mock Provider mode.
* Existing GitHub Pages deployment remains available during validation.

# 21. Cost-Optimization Decisions

Initial cost controls:

* Use S3 for static object storage.
* Use CloudFront caching to reduce repeated origin requests.
* Use compressed frontend assets.
* Avoid EC2 or container hosting for the static frontend.
* Avoid unnecessary Lambda@Edge functions.
* Avoid WAF until justified by production risk and traffic.
* Use a suitable CloudFront price class for the expected audience.
* Use lifecycle rules for old S3 object versions if versioning creates unnecessary storage growth.
* Apply long cache durations to hashed assets.
* Apply short cache duration to `index.html`.

# 23. Rollback Plan

GitHub Pages remains the current public hosting platform during the AWS validation period.

If the CloudFront deployment fails:

1. Do not change Namecheap DNS.
2. Keep GitHub Pages active.
3. Inspect CloudFormation stack events.
4. Correct the S3, OAC, bucket-policy or CloudFront configuration.
5. Redeploy the frontend stack.
6. Upload the frontend build again.
7. Retest through the CloudFront domain.

If a bad frontend build is uploaded:

1. Rebuild the last known-good commit.
2. Synchronize the correct `dist/` directory to S3.
3. Invalidate CloudFront.
4. Verify the application.

S3 versioning may also be used to recover earlier object versions.

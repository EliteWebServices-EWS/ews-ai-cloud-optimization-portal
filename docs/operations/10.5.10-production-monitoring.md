# Objective

Implement centralized production monitoring and observability for the SISU'M platform using Amazon CloudWatch, CloudWatch Alarms, and Amazon SNS.

This sprint provides real-time visibility into the health of the production frontend and backend while enabling proactive alerting when issues occur.

# CloudWatch Alarms

The following production alarms were configured.

## Lambda Errors

Alarm Name

```text
sisum-production-lambda-errors
```

Triggers when one or more Lambda errors occur within five minutes.

---

## Lambda Throttles

Alarm Name

```text
sisum-production-lambda-throttles
```

Triggers whenever Lambda experiences throttling.

---

## Lambda Duration

Alarm Name

```text

```

# Operational Response Procedure

If an alarm is triggered:

1. Open the CloudWatch Dashboard.
2. Identify the affected service.
3. Review Lambda metrics.
4. Review API Gateway metrics.
5. Review CloudFront metrics.
6. Open CloudWatch Logs.
7. Verify backend health endpoint.
8. Verify production website.
9. Verify dashboard functionality.
10. Redeploy or roll back if required.
11. Document the incident.

---

# Known Limitations

Current implementation:

- CloudFront metrics may take several minutes before appearing.
- S3 storage metrics are updated daily.
- No synthetic canary monitoring has been configured.
- No Slack, Microsoft Teams, or PagerDuty integration has been configured.
- Alarm thresholds may require adjustment after observing production traffic over time.

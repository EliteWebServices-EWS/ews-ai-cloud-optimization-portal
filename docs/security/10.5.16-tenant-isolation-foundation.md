## Objective

Make the SISU'M platform tenant-aware before connecting real customer AWS accounts. Every user-facing read and write must enforce the authenticated user's tenant boundary.

## Scope

- Trusted tenant identity from Cognito `custom:tenantId` claim
- Request security context with centralized tenant resolution
- Tenant ID on workflow, report, execution, verification, learning, and audit records
- Tenant-scoped in-memory stores (workflow, report, learning)
- Cross-tenant access returns safe 404 without leaking ownership
- Security audit events for tenant claim issues and cross-tenant attempts
- Compatibility and strict enforcement modes
- Cognito custom attribute template update
- Monitoring metric filter and alarm for `tenant.access_denied`
- Comprehensive unit and integration tests

## Out of scope

Real AWS AssumeRole integration, customer onboarding, billing, per-tenant branding/domains, invitation emails, multiple Cognito pools, Organizations integration, WAF, real provider execution, write permissions to customer AWS accounts.

## Threat model

| Threat | Mitigation |
|--------|------------|
| Tenant ID spoofing via headers/body/query | Strip `x-tenant-id`; never read `tenantId` from client input |
| Header injection (`x-sisum-tenant-id`) | Lambda strips client internal headers before attaching trusted claims |
| Cross-tenant direct object reference | Repository lookups require `tenantId`; composite store keys |
| Cross-tenant report/workflow access | Tenant-scoped `get`/`list`; 404 on mismatch |
| Cross-tenant execution | Execution context carries trusted `tenantId` from workflow |
| Audit-query abuse | Admin queries scoped to authenticated tenant; `tenantId` query param rejected |
| Cognito tenant reassignment | Document admin-only assignment; mutable attribute with operational risk noted |
| Missing claim fallback abuse | Compatibility mode logs `tenant.claim_missing` and `tenant.fallback_used` |
| Repository called without tenant | Store interfaces require `tenantId` on all lookups |
| Logging another tenant's identifiers | Cross-tenant denials return generic 404; audit uses request tenant only |

See also: [tenant-boundary-threat-model.md](./tenant-boundary-threat-model.md)

## Trusted tenant source

1. Administrator assigns `custom:tenantId` on the Cognito **user profile** (not client-writable).
2. **Pre Token Generation Lambda (V2_0)** reads `custom:tenantId` and injects `tenant_id` into the **access token**.
3. SPA sends the access token to API Gateway (`frontend/dashboard/src/api/client.ts`).
4. API Gateway JWT authorizer validates the access token (`backend/template.yaml`).
5. `backend/lambda.ts` rejects non-access tokens and copies `tenant_id` â†’ `x-sisum-tenant-id`.
6. `requireTenantContext()` resolves the final tenant boundary.

**Important:** Cognito does **not** automatically include `custom:tenantId` in access tokens. Reading `custom:tenantId` directly from the validated access token was incorrect and has been corrected to use `tenant_id`.

**Never trusted:** request body, query string, route parameters, `x-tenant-id`, client `x-sisum-tenant-id` (stripped at Lambda; only API Gateway-validated claims are copied).

## Cognito custom attribute strategy

Added to `infrastructure/auth/template.yaml`:

```yaml
- Name: tenantId
  AttributeDataType: String
  Mutable: true
  Required: false
  StringAttributeConstraints:
    MinLength: "1"
    MaxLength: "64"
```

Maps to user profile attribute `custom:tenantId`. The Pre Token Generation trigger validates this value and injects `tenant_id` into access tokens. Attribute is **mutable** to support administrator reassignment; reassignment is a privileged operation with security implications documented in the tenant assignment runbook.

## Access-token customization (Pre Token Generation V2_0)

Cognito does **not** automatically place `custom:tenantId` in access tokens. The auth stack deploys an **inline Lambda** (single source of truth in `infrastructure/auth/template.yaml` `Code.ZipFile`).

### Trigger behavior

| Profile attribute | Trigger action | Access token |
|-------------------|----------------|--------------|
| Valid `custom:tenantId` | Inject claim | `tenant_id` set |
| Missing attribute | Return event unchanged | No `tenant_id` â€” compatibility fallback applies |
| Invalid attribute | Return event unchanged | No `tenant_id` â€” compatibility fallback applies |

Validation pattern (same as backend):

```text
/^(?:[a-z0-9]|[a-z0-9][a-z0-9-]{0,62}[a-z0-9])$/
```

Length 1â€“64. The trigger reads **only** `event.request.userAttributes['custom:tenantId']`. It does not trust `clientMetadata`, headers, body, or query parameters. It preserves existing access-token claims and scopes when adding `tenant_id`.

### CloudFormation deployment ordering

To avoid Cognito rejecting trigger attachment, resources are created in this order:

```text
SisumPreTokenGenerationRole
  â†’ SisumPreTokenGenerationFunction
  â†’ SisumPreTokenGenerationPermission (SourceAccount scoped)
  â†’ SisumUserPool (PreTokenGenerationConfig V2_0 only)
  â†’ SisumUserPoolClient / domain / groups
```

The Lambda permission uses:

```yaml
Principal: cognito-idp.amazonaws.com
SourceAccount: !Ref AWS::AccountId
```

This allows Cognito in the same AWS account to invoke the trigger **before** the user pool exists, avoiding a circular dependency with `SourceArn: !GetAtt SisumUserPool.Arn`.

**Follow-up hardening (optional):** After the pool is stable, a second deployment may replace `SourceAccount` with a pool-specific `SourceArn` if operational review confirms no circular dependency.

The legacy `PreTokenGeneration` property is **not** used â€” only `PreTokenGenerationConfig` with `LambdaVersion: V2_0`.

## Compatibility and strict mode

| Mode | Env var | Missing claim behavior |
|------|---------|------------------------|
| `compatibility` (default) | `TENANT_ENFORCEMENT_MODE=compatibility` | Fallback to `DEFAULT_TENANT_ID`; audit `tenant.claim_missing` + `tenant.fallback_used` |
| `strict` | `TENANT_ENFORCEMENT_MODE=strict` | 403 `TENANT_REQUIRED` unless `TENANT_FALLBACK_ENABLED=true` |

Production default remains **compatibility** until access tokens are verified to contain `tenant_id` after fresh login. **Strict mode must remain disabled** until that verification completes. Invalid or missing profile attributes result in no `tenant_id` claim â€” compatibility mode falls back to `DEFAULT_TENANT_ID` with audit events.

**Deployment note:** Cognito custom schema additions may not apply to existing user pools via stack update. If the stack update cannot add the attribute, perform manual schema migration before enabling strict mode.

## Workflow isolation

- `executeWorkflow({ tenantId, ... })` stamps tenant on metadata and context
- `getWorkflow(tenantId, workflowId)` and `getWorkflowStatus(tenantId, workflowId)` require tenant
- Cross-tenant lookup returns `undefined` â†’ route returns 404

## Report isolation

- Report generation inherits tenant from workflow record
- Cached report lookup is tenant-scoped
- List reports filtered by authenticated tenant

## Execution and verification isolation

- Execution simulator stamps `metadata.tenantId` from workflow context
- Verification reports include `tenantId`
- Learning records stamped at save time

## Audit isolation

- Audit events enriched with trusted `tenantId`
- `tenant.access_denied` includes audit-only `resourceTenantId` (owning tenant of the requested resource; never returned to clients)
- Admin `/admin/audit-events` uses authenticated tenant only
- Client `tenantId` query parameter rejected with `TENANT_QUERY_FORBIDDEN`

### `tenant.access_denied` audit payload

| Field | Meaning |
|-------|---------|
| `tenantId` | Requesting tenant (from trusted access token) |
| `resourceTenantId` | Tenant that owns the requested resource (audit-only) |
| `resource.type` | Resource type (e.g. `workflow`, `report`) |
| `resource.id` | Resource identifier |
| `actor.userId` | Authenticated user |
| `requestId` | Request correlation |
| `correlationId` | Cross-service correlation |

## Error behavior

| Scenario | Response | Audit event |
|----------|----------|-------------|
| Cross-tenant resource lookup | 404 NOT_FOUND | `tenant.access_denied` |
| Missing claim (strict) | 403 TENANT_REQUIRED | `tenant.claim_missing` |
| Missing claim (compatibility) | 200 with fallback tenant | `tenant.claim_missing`, `tenant.fallback_used` |
| Invalid claim format (strict) | 403 INVALID_TENANT_CLAIM | `tenant.claim_missing` |

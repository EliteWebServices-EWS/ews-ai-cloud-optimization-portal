# Sprint 10.5.13 — Audit Logging and User Activity Tracking

## Status

**In Progress**

## Objective

Implement structured, security-focused audit logging for the SISU'M Cloud Optimization Decision Platform.

The audit system must record important authenticated user actions without storing passwords, access tokens, authorization headers, or other secrets.

This sprint builds on the Amazon Cognito authentication and role-based access control completed in Sprints 10.5.11 and 10.5.12.

---

## Business Purpose

The platform must be able to answer:

- Who performed an action?
- What action was attempted?
- When did the action occur?
- Which role did the user have?
- Which resource or workflow was involved?
- Was the action successful, denied, or failed?
- Which request ID can be used to trace related logs?
- How long did the request take?

These records support:

- Operational troubleshooting
- Security investigations
- Accountability
- Incident response
- Change tracking
- Future SOC 2 readiness
- Future ISO 27001 readiness

This sprint improves audit readiness but does not by itself certify the platform for any compliance framework.

## Audit Event Model

Every audit event should use a consistent structure.

```json
{
  "type": "AUDIT",
  "version": "1.0",
  "timestamp": "2026-07-19T12:00:00.000Z",
  "eventName": "workflow.completed",
  "outcome": "success",
  "severity": "INFO",
  "requestId": "req-example",
  "correlationId": "corr-example",
  "actor": {
    "userId": "cognito-user-id",
    "email": "user@example.com",
    "groups": ["analyst"]
  },
  "request": {
    "method": "POST",
    "path": "/api/v1/workflows/run",
    "sourceIp": "203.0.113.10",
    "userAgent": "Mozilla/5.0"
  },
  "resource": {
    "type": "workflow",
    "id": "wf-example"
  },
  "metadata": {
    "durationMs": 23,
    "statusCode": 201
  }
}
```

Fields that do not apply may be omitted or set to `null`.

---

## Required Audit Fields

| Field | Purpose |
|---|---|
| `type` | Distinguishes audit logs from normal application logs |
| `version` | Audit-schema version |
| `timestamp` | UTC event time |
| `eventName` | Machine-readable action name |
| `outcome` | `success`, `denied`, or `failure` |
| `severity` | `INFO`, `WARN`, or `ERROR` |
| `requestId` | Unique ID for one HTTP request |
| `correlationId` | ID shared across related operations |
| `actor.userId` | Cognito subject or username |
| `actor.email` | User email when available |
| `actor.groups` | Cognito roles assigned to the user |
| `request.method` | HTTP method |
| `request.path` | Requested API path |
| `request.sourceIp` | Client IP supplied by API Gateway |
| `request.userAgent` | Client user agent |
| `resource.type` | Workflow, report, execution, or authorization |
| `resource.id` | Related workflow or report ID |
| `metadata.durationMs` | Processing duration |
| `metadata.statusCode` | Resulting HTTP status |

## Security and Privacy Requirements

Audit logging must never record:

- Passwords
- Temporary passwords
- Access tokens
- ID tokens
- Refresh tokens
- Authorization headers
- Cookies
- Session-storage contents
- AWS secret access keys
- Full request bodies by default
- Sensitive customer payloads
- Unredacted secrets

Audit logging may record:

- Cognito user ID
- User email
- Cognito groups
- Request path
- HTTP method
- Source IP
- User agent
- Workflow ID
- Report ID
- Result and duration
- Safe resource identifiers

## Correlation-ID Design

### Request ID

A request ID identifies one API request.

Preferred sources, in order:

1. Existing API Gateway request ID
2. Existing trusted request ID header
3. Generated SISU'M request ID

Example:

```text
req-mrq67noj
```

### Correlation ID

A correlation ID groups related requests or workflow operations.

Preferred sources, in order:

1. Valid client-provided `x-correlation-id`
2. Workflow ID when a workflow exists
3. Generated correlation ID

Example:

```text
corr-mrq67noj
```

The response should return:

```http
x-request-id: req-example
x-correlation-id: corr-example
```

## Planned Middleware Order

The Express middleware order is important.

```text
1. CORS middleware
2. JSON-body parser
3. Request-context middleware
4. Request completion/error audit middleware
5. API routes
6. RBAC middleware within protected routes
7. Route handlers
```

The request-context middleware must execute before RBAC so authorization denials contain:

- request ID
- correlation ID
- actor identity
- path
- source IP
- user agent

## CloudWatch Logs Insights Examples

### Find authorization denials

```sql
fields @timestamp, eventName, actor.email, actor.groups, request.path
| filter type = "AUDIT"
| filter eventName = "authorization.denied"
| sort @timestamp desc
```

### Find workflow completions

```sql
fields @timestamp, actor.email, resource.id, metadata.durationMs
| filter type = "AUDIT"
| filter eventName = "workflow.completed"
| sort @timestamp desc
```

### Find failed audited actions

```sql
fields @timestamp, eventName, actor.email, request.path, metadata
| filter type = "AUDIT"
| filter outcome = "failure"
| sort @timestamp desc
```

## Risks and Controls

| Risk | Control |
|---|---|
| Tokens accidentally logged | Never log headers or session objects |
| Excessive CloudWatch costs | Log security and business events, not entire payloads |
| Personal data overcollection | Restrict actor data to ID, email, and groups |
| Duplicate events | Use request ID and event name for correlation |
| Missing identity | Record `identity.missing` without inventing a user |
| User-supplied spoofed identity | Trust only Lambda-generated internal identity headers |
| Audit logger breaks requests | Audit logging must fail safely and not expose secrets |

---

## Rollback Plan

If audit logging causes production errors:

1. Revert the audit-logging commit.
2. Run backend tests.
3. Run `sam validate --lint`.
4. Run `sam build`.
5. Deploy the prior known-good backend.
6. Confirm:

```text
GET /api/v1/health → HTTP 200
```

7. Revalidate RBAC:

```text
viewer → 403 for analysis
analyst → analysis succeeds
admin → analysis succeeds
```

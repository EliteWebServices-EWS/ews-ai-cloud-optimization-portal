# Sprint 10.5.15 — Production Security Hardening

## Objective

Strengthen the SISU'M production security foundation through privileged-user MFA, authentication review, CORS hardening, security headers, request protection, API throttling, secrets classification, IAM least privilege, encryption verification, monitoring improvements, dependency review, and operational runbooks — without weakening existing controls.

## Scope

- Amazon Cognito TOTP MFA enablement (optional platform policy)
- JWT identity extraction hardening
- Production CORS allowlist enforcement
- API security response headers
- JSON body size limits and input validation
- API Gateway throttling and Lambda concurrency bounds
- IAM runtime policy tightening
- CloudWatch security monitoring additions
- Security documentation, threat model, and compliance readiness mappings
- Unit and integration test coverage for security controls

## Out-of-scope items

- **AWS WAF** — deferred to a pre-customer-launch edge-security sprint
- Real AWS provider activation (`PROVIDER_MODE` remains `mock`)
- Customer-managed KMS keys
- Secrets Manager resources without actual secrets
- Group-specific Cognito MFA enforcement via custom Lambda triggers
- SOC 2 or ISO 27001 certification

- Cognito Authorization Code + PKCE authentication

## Authentication changes

### MFA decision

**Chosen policy:** Cognito `MfaConfiguration: OPTIONAL` with `SOFTWARE_TOKEN_MFA` (TOTP authenticator app).

| Requirement | Implementation | Status |
|-------------|----------------|--------|
| Admin MFA | TOTP enabled; operational enrollment required before granting admin group membership | Partially enforced |
| Analyst MFA | Same operational procedure recommended; not natively enforced by group | Partially enforced |
| Viewer MFA | Optional at current stage | Optional |
| SMS MFA | Not enabled | N/A |

**Limitation:** Amazon Cognito does not natively enforce MFA by user pool group without custom `DefineAuthChallenge` / pre-authentication Lambda triggers. Requiring MFA platform-wide (`MfaConfiguration: ON`) would also affect viewers. This sprint adopts **optional TOTP with operational enforcement** for privileged users rather than an unreliable group-specific workaround.

**Operational requirement:** Admin and analyst accounts must enroll TOTP before production use. Access reviews must verify MFA status via Cognito console or AWS CLI.

### Password and account recovery (unchanged)

- Minimum length 12 with complexity requirements
- Admin-created users only (`AllowAdminCreateUserOnly: true`)
- Verified email account recovery
- Email verification required before email attribute updates

## JWT review

| Control | Layer | Status |
|---------|-------|--------|
| Issuer validation | API Gateway JWT authorizer | Implemented |
| Audience validation | API Gateway JWT authorizer | Implemented |
| Expiration validation | API Gateway JWT authorizer | Implemented |
| Token use validation | Lambda strips non-`access` tokens before identity headers | Implemented |
| Subject extraction | Lambda → `x-sisum-user-id` from `sub` | Implemented |
| Group extraction | Lambda → parsed in `backend/auth/identity.ts` | Implemented |
| Email extraction | Normalized; empty strings become `null` | Fixed |
| Unknown role rejection | `ROLE_UNRECOGNIZED` audit event + 403 | Implemented |
| Missing identity | `IDENTITY_MISSING` audit event + 401 | Implemented |

Cryptographic JWT verification remains at API Gateway. Lambda trusts only API Gateway-validated claims and removes client-supplied identity headers.

## CORS policy

### Production allowlist

- `https://elitewebservices.org`
- `https://www.elitewebservices.org`

Configured in:

- `backend/security/cors.ts` (Express middleware with `Vary: Origin`)
- `backend/template.yaml` API Gateway `CorsConfiguration`
- `CORS_ALLOWED_ORIGINS` environment variable

### Removed from production

- Wildcard `*` (except explicit local dev via `CORS_ORIGIN=*`)
- `https://elitewebservices-ews.github.io`
- CloudFront distribution origin (documented fallback constant only; not in production allowlist)

### Local development

Set `CORS_ALLOW_LOCAL=true` or `NODE_ENV=development` to include `http://localhost:5173`.

## Security headers

Custom middleware in `backend/security/security-headers.ts` (no `helmet` dependency):

| Header | Value |
|--------|-------|
| Strict-Transport-Security | `max-age=31536000; includeSubDomains` |
| X-Content-Type-Options | `nosniff` |
| X-Frame-Options | `DENY` |
| Referrer-Policy | `strict-origin-when-cross-origin` |
| Permissions-Policy | `camera=(), microphone=(), geolocation=()` |
| Content-Security-Policy | `default-src 'none'; frame-ancestors 'none'` |
| Cache-Control | `no-store, no-cache, must-revalidate, private` |

## Request size limits

- JSON body limit: **256 KB** (`backend/security/request-limits.ts`)
- Malformed JSON → `400 INVALID_JSON`
- Oversized body → `413 PAYLOAD_TOO_LARGE`
- Stack traces are not exposed in error responses

## Input validation

Centralized validators in `backend/security/request-validation.ts`:

- Workflow run: plugin, mode, resourceId, region
- Report generate: workflowId
- Audit query limits and pagination tokens (existing + extended patterns)
- Bounded string lengths and identifier character sets

## API throttling

| Control | Setting | Rationale |
|---------|---------|-----------|
| API Gateway burst | 100 requests | Absorb short dashboard bursts |
| API Gateway steady rate | 50 req/s | Prevent sustained abuse |
| Lambda reserved concurrency | 25 | Cap parallel execution cost |

429 responses originate at API Gateway before Lambda invocation. HTTP API `4xx` metrics include throttling.

## Known limitations

- MFA is optional at the Cognito platform level; privileged-user MFA relies on operational enforcement
- No native group-specific MFA without custom Lambda triggers
- API Gateway 429 is aggregated in `4xx` metrics
- Single-tenant MVP — no full multi-tenancy isolation
- In-memory Express rate limiting not used (by design — not durable in Lambda)

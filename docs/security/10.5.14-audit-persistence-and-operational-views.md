# Sprint 10.5.14 — Audit Persistence and Operational Views

## Status

**Complete (implementation and local validation)**

## Objective

Persist important audit events to DynamoDB in addition to existing CloudWatch structured logging, and provide admin-only operational methods to retrieve and investigate those events.

This sprint builds on [Sprint 10.5.13 — Audit Logging](./10.5.13-audit-logging.md).

---

## Architecture

```text
API request
   |
   v
Existing audit event builder (buildAuditEvent / writeAuditEvent)
   |
   +--> Structured JSON to CloudWatch (console.info/warn/error)
   |
   +--> scheduleAuditPersistence(req, event)
           |
           v
        auditPersistenceFlushMiddleware (awaits pending writes before res.end)
           |
           v
        DynamoDB PutItem (sisum-audit-<environment>)
```

### Failure handling decision

**Option B — resilient design**

Audit persistence failures do **not** fail the business request. Failures are logged as structured `audit.persistence_failed` events to CloudWatch and can trigger monitoring alarms. No secrets from the failed event are included in the failure log.

---

## Data flow

1. Route or middleware calls `writeAuditEvent()` for immediate CloudWatch output.
2. The same event is passed to `scheduleAuditPersistence(req, event)`.
3. Before the HTTP response is sent, `auditPersistenceFlushMiddleware` waits for pending DynamoDB writes via `Promise.allSettled`.
4. Administrators query persisted records through `GET /api/v1/admin/audit-events`.
5. Admin searches emit `audit.search_performed` to CloudWatch only (not persisted, avoiding recursion).

## DynamoDB table design

| Property | Value |
|---|---|
| Logical resource | `SisumAuditTable` |
| Physical name | `sisum-audit-<environment>` |
| Billing | `PAY_PER_REQUEST` |
| Encryption | AWS owned key (`SSEEnabled: true`) |
| PITR | Enabled |
| TTL attribute | `expiresAt` (Unix epoch seconds) |
| Deletion policy | `Retain` |
| Update replace policy | `Retain` |

### Key schema

| Key | Pattern | Example |
|---|---|---|
| `pk` (partition) | `TENANT#<tenantId>` | `TENANT#sisum-default` |
| `sk` (sort) | `AUDIT#<timestamp>#<eventId>` | `AUDIT#2026-07-19T12:00:00.000Z#uuid` |

No GSIs were added. Tenant-scoped queries use the primary key with optional `FilterExpression` for bounded admin searches.

---

## Tenant strategy

Full multi-tenancy is **not** implemented.

| Phase | Strategy |
|---|---|
| MVP (this sprint) | All records use `DEFAULT_TENANT_ID` (`sisum-default`) from Lambda environment |
| Future | Replace with trusted tenant claim from Cognito custom attributes or tenant registry |

**Rules**

- Never accept `tenantId` from client request body or query parameters.
- Admin queries are always scoped to the resolved tenant before DynamoDB access.

---

## Retention and TTL

| Setting | Default | Environment variable |
|---|---|---|
| Retention | 365 days | `AUDIT_RETENTION_DAYS` |
| TTL field | `expiresAt` | Calculated at write time |

DynamoDB TTL automatically deletes expired items. CloudWatch log retention remains 14 days (unchanged).

---

## IAM permissions

Policy `sisum-audit-dynamodb-<environment>` attached to `SisumLambdaExecutionRole`:

```yaml
Action:
  - dynamodb:PutItem
  - dynamodb:GetItem
  - dynamodb:Query
Resource:
  - <SisumAuditTable.Arn>
```

No wildcard table permissions. No `dynamodb:Scan`.

## Persisted vs CloudWatch-only events

| Event | CloudWatch | DynamoDB |
|---|---|---|
| `request.started` | Yes | No (volume control) |
| `request.completed` | Yes | No (volume control) |
| `request.failed` | Yes | Yes |
| Authorization / workflow / report / execution events | Yes | Yes |
| `audit.search_performed` | Yes | No (recursion prevention) |
| `audit.persistence_failed` | Yes | Yes |

## Admin API

### Endpoint

```http
GET /api/v1/admin/audit-events
```

### Authorization

| Role | Access |
|---|---|
| `admin` | Allowed |
| `analyst` | Denied (403) |
| `viewer` | Denied (403) |

### Query parameters

| Parameter | Description |
|---|---|
| `eventName` | Filter by audit event name |
| `outcome` | `started`, `success`, `failure`, `denied` |
| `actorUserId` | Cognito subject |
| `workflowId` | Workflow identifier |
| `requestId` | Request identifier |
| `correlationId` | Correlation identifier |
| `from` | ISO-8601 UTC lower bound |
| `to` | ISO-8601 UTC upper bound |
| `limit` | Page size (max 100, default 50) |
| `nextToken` | Pagination token |

Returns `503` when `AUDIT_PERSISTENCE_ENABLED=false`.

## CloudWatch Logs Insights queries

These queries match the actual JSON structure emitted by the backend (`category = "audit"`).

### Authorization denials

```sql
fields @timestamp, eventName, actor.email, actor.roles, path, statusCode, correlationId
| filter category = "audit"
| filter eventName = "authorization.denied"
| sort @timestamp desc
| limit 100
```

### Missing identities

```sql
fields @timestamp, eventName, actor.email, path, statusCode, requestId
| filter category = "audit"
| filter eventName = "identity.missing"
| sort @timestamp desc
| limit 100
```

### Failed workflows

```sql
fields @timestamp, eventName, actor.email, workflowId, reason, errorCode, durationMs
| filter category = "audit"
| filter eventName = "workflow.failed"
| sort @timestamp desc
| limit 100
```

### Failed report generation

```sql
fields @timestamp, eventName, actor.email, workflowId, reportId, reason, errorCode
| filter category = "audit"
| filter eventName = "report.generation_failed"
| sort @timestamp desc
| limit 100
```

### Failed simulations

```sql
fields @timestamp, eventName, actor.email, workflowId, executionId, reason, errorCode
| filter category = "audit"
| filter eventName = "execution.simulation_failed"
| sort @timestamp desc
| limit 100
```

### Audit persistence failures

```sql
fields @timestamp, eventName, failedEventName, failedEventId, reason, requestId
| filter category = "audit"
| filter eventName = "audit.persistence_failed"
| sort @timestamp desc
| limit 100
```

### Activity by actor

```sql
fields @timestamp, eventName, actor.email, actor.roles, path, outcome
| filter category = "audit"
| filter actor.email = "admin@example.com"
| sort @timestamp desc
| limit 100
```

### Activity by correlation ID

```sql
fields @timestamp, eventName, actor.email, path, outcome, statusCode
| filter category = "audit"
| filter correlationId = "corr-example"
| sort @timestamp desc
| limit 100
```

### Activity by workflow ID

```sql
fields @timestamp, eventName, actor.email, workflowId, outcome, durationMs
| filter category = "audit"
| filter workflowId = "wf-example"
| sort @timestamp desc
| limit 100
```

### Slow requests (> 5000 ms)

```sql
fields @timestamp, eventName, path, durationMs, statusCode, actor.email
| filter category = "audit"
| filter eventName = "request.completed" or eventName = "request.failed"
| filter durationMs > 5000
| sort durationMs desc
| limit 100
```

### HTTP 5xx requests

```sql
fields @timestamp, eventName, path, statusCode, actor.email, reason, errorCode
| filter category = "audit"
| filter statusCode >= 500
| sort @timestamp desc
| limit 100
```

| Alarm | Reason deferred |
|---|---|

| Repeated audit query failures | Low volume expected initially; monitor via `audit.search_performed` failures in Logs Insights |

## Security considerations

### No-secret policy

Never persisted or logged:

- Authorization headers
- JWTs / access / refresh / ID tokens
- Passwords / cookies
- AWS credentials
- Complete request bodies

### Controls added

- Admin-only audit API (`ADMIN_ROLES`)
- Tenant scoping before query
- Bounded pagination (max 100)
- Input validation on all query filters
- Least-privilege DynamoDB IAM
- Encryption at rest and PITR on audit table
- Search audit events excluded from persistence (recursion prevention)

2. Run `npm ci`, `npm run build`, `npm test`, `sam validate --lint`, `sam build`.
3. Redeploy the prior backend stack.
4. Set `AuditPersistenceEnabled=false` as a hotfix if DynamoDB issues occur without redeploying code.
5. DynamoDB table uses `DeletionPolicy: Retain` — table data survives stack rollback.
6. Confirm:
   - `GET /api/v1/health` → 200
   - CloudWatch audit logging still works
   - RBAC unchanged

## Cost considerations

| Driver | Impact |
|---|---|
| Audit write volume | On-demand write units per persisted event |
| Retention duration | Storage until TTL expiration |
| GSI writes | None (no GSIs) |
| CloudWatch ingestion | Unchanged (all events still logged) |
| CloudWatch retention | 14 days (unchanged) |
| Admin query volume | On-demand read units, bounded by pagination |
| PITR | Small additional storage cost on audit table |

Cost controls: on-demand billing, TTL, no streams, no global tables, Query-only access pattern.

## Known limitations

- Single default tenant (`sisum-default`); not full multi-tenancy
- No audit search frontend page
- No S3 archival
- `request.started` / `request.completed` not persisted (CloudWatch only)
- Filter expressions on Query may consume read capacity beyond returned items
- External IAM role must exist before deploy (`SisumLambdaExecutionRole`)
